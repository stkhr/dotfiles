# AI駆動開発 共通ガイドライン

## 最優先原則(事実性)

- 実行していない作業を完了として報告しない。ファイル変更・commit・push・PR作成・外部APIの結果は、実行直後に実出力(`git status -sb` / `gh pr view` / コマンドのJSON出力)で確認できたものだけ「完了/成功」と書く。確認できないものは「未確認」と明示する
- ツールの出力が空・壊れている時に、結果を推測で埋めない。再実行して実出力を得てから報告する

## Git Workflow

- コンベンショナルコミット形式を使用: `feat:`, `fix:`, `docs:`, `test:`, `refactor:`, `chore:`
- コミットメッセージは英語、PRのタイトル・本文は日本語で記述

## PR作成の方針

- コミットを伴う依頼は常にフィーチャーブランチを切ってPR化する(1行修正でも)。例外はユーザーが明示的に「直接コミットしてよい」「PR不要」と指示した場合のみ
- 委譲を伴う実装タスクの既定の終点は Draft PR(作成の事前承認は不要)。作成後に PR URL・本文要旨・残レビュー所見を共有する
- テンプレート探索・本文フォーマット・Draft PRフローの手順は `pr-creation` スキルに従う

## 破壊的・外部影響操作の確認ハードゲート

git/gh の破壊的操作(force push / `--no-verify` / ブランチ削除 / PRの close・comment・merge・ready / 非Draft PR作成)は hook が確認プロンプトを強制する。hook が検知できない以下の操作も、実行前に「文面提示 → ユーザー承認 → 実行」を経ること:

- 外部チャネル(チャット・チケットシステム等)への投稿・状態変更
- 既存ファイルの大量削除 / 一括リネーム

「ロールバックの急ぎ」「CIを通すための一時回避」を理由に独断実行しない。回避策が必要ならまずユーザーに確認する。

## GitHub CLI (`gh`)

GitHub関連の操作はすべて `gh` コマンドを使用する(MCPサーバーは使用しない)。

## CI/CD

- PR作成前に必ず `superpowers:requesting-code-review` スキルを実行し、独立したレビューエージェントの承認を得る(自己評価での代替不可)。Critical な指摘は修正してから再レビュー
- Draft PR では Important・Minor を本文の「レビュー所見」に記載して人間の判断に委ねる。非Draft(Ready)PR を作成する場合は Important も修正してから
- CI/CD が設定されている場合、PR作成後 `gh pr checks` でステータス確認までが作業完了。CI失敗は即修正(放置・無視は禁止)

## crit によるローカルレビュー(設計・コード)

- 対話セッションで設計/プランファイルを生成・更新したら、実装前に crit レビューを通す(順序: 設計の crit レビュー → 実装 → requesting-code-review → Draft PR)
- tmux 内: `crit review <file> --detach --wait`(Bash timeout は 600000ms)→ `crit status <file>` のコメントを反映。tmux 外 / headless ではブロッキング起動せず、手動実行をユーザーに依頼する
- TUI を開いている間は対象ファイルを編集しない

## コードレビュー指摘の受け止め方

- 可読性を下げる方向の指摘(条件式の否定形への反転、抽象化の追加、将来仮定への対応)はそのまま適用しない。該当箇所・指摘要約・代替案(適用しない案を含む)を提示してユーザーの判断を仰ぐ
- 「CIが通った」「レビューエージェントが承認した」を設計判断の正当性の根拠としない。人間の判断を上書きしない
- 指摘への反論コメントをPR/Issueに投稿する場合は、文面の承認を得てから投稿する

## 設計と実装の一貫性

- 技術的制約により設計(How)を変更した場合、PR作成前に設計ドキュメントを実装に合わせて更新する。What(実現したいこと)は変えない
- 乖離が残る場合はPRの説明に変更理由を記載する

## エラーハンドリング

- エラーの抑制(`@ts-ignore`、try-catchの握りつぶし)ではなく根本原因を修正する
- 関連が薄く見えるエラーでも必ず解決する

## 外部API・クラウド設定の事前確認

クラウドAPI・外部SaaSを扱うコードを書く前に、必ず一次情報(公式ドキュメント or 実機のCLI/APIレスポンス)で以下2軸を確認すること:

- **識別子と命名規則**: action prefix・サービス識別子・suffix・正規表現制約
- **スキーマと制約値**: リクエスト/レスポンス構造・必須フィールド・プラットフォーム推奨設定との整合

「実装 → 実行時エラー → 段階的修正」はユーザーのレビュー負荷を増やす。実装着手前の確認を優先する。

### 仮説の検証ルール

- 同じ問題で仮説が3回外れたら、レイヤを1つ上げて疑う(ミドルウェア → アプリ → DB / ネットワーク → OS)
- 推測のまま設定値・コード値を確定させない。実機検証(CLIワンライナーをユーザーに提供して結果を貰う等)を挟む

## コメント・記述の簡潔さ(デフォルト)

- コードコメントは「なぜそうするか」「非自明な制約・前提・落とし穴」のみ。処理を逐語的になぞるコメント、型・変数名から自明な説明は書かない
- PR本文・コミットメッセージは diff で分かる「何を」を最小化し「なぜ」に絞る。前置き・定型句の繰り返しを避ける

## セキュリティ

- APIキー、パスワード等は環境変数で管理(ハードコード禁止)
- 日常の脆弱性検出は `security-guidance` plugin が自動実行する。能動的なセキュリティ強化(脅威モデリング、依存関係スキャン等)は `/security-hardening` スキルを使用

## タスク指示のフォーマット(委譲型)

Goal / Constraints / Acceptance criteria が揃っている場合はそれに従い、揃っていない場合も細かい手順確認は最小限にとどめ自律的に進める。コミットを伴う実装タスクの終点は原則 Draft PR の作成とリンク共有。

- 例外: 人間が在席する対話セッションで設計/プランファイルを生成した場合のみ、実装前に crit レビューを差し込む。headless 自走時は Draft PR を人間レビューのゲートとする

## Subagent の使用指針

- 複数ファイルへの並列処理、互いに独立した複数タスクで積極活用する
- それ以外での不要な Subagent 呼び出しは避ける(過剰な委譲は性能低下のリスク)

## Terraform スタイル規範

- `count` / `for_each` の三項演算子は肯定形を使う(OK: `var.env_name == "stg" ? 1 : 0` / NG: `var.env_name != "prd" ? 1 : 0`)
- `terraform fmt` は変更ファイル単位で実行する(`-recursive` でスコープ外まで整形しない)

## Compaction preferences

コンテキスト圧縮時は、以下を必ず保持:

- 変更したファイルの一覧
- 実行したテストコマンドとその結果
- 未完了のタスク
