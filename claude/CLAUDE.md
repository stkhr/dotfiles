# AI駆動開発 共通ガイドライン

## 最優先原則(事実性)

- 実行していない作業を完了として報告しない。ファイル変更・commit・push・PR作成・外部APIの結果は、実行直後に実出力(`git status -sb` / `gh pr view` 等)で確認できたものだけ「完了/成功」と書く。確認できないものは「未確認」と明示する
- ツールの出力が空・壊れている時に、結果を推測で埋めない。再実行して実出力を得てから報告する

## Git / PR ワークフロー

- コンベンショナルコミット形式(`feat:` `fix:` `docs:` `test:` `refactor:` `chore:`)。コミットメッセージは英語、PRタイトル・本文は日本語
- GitHub関連の操作はすべて `gh` コマンドを使用する(MCPサーバーは使用しない)
- ファイル編集を伴うタスクは着手前に EnterWorktree で worktree に入る。例外は既に worktree 内にいる場合と、ユーザーが「worktree 不要」と明示した場合のみ。worktree はリポジトリ配下の `.claude/worktrees/` に置く
- コミットを伴う依頼は常にフィーチャーブランチを切ってPR化する(1行修正でも)。例外はユーザーが「直接コミットしてよい」「PR不要」と明示した場合のみ
- 委譲を伴う実装タスクの既定の終点は Draft PR(作成の事前承認は不要)。テンプレート探索・本文フォーマット・レビュー→作成→共有→CI確認の手順は `pr-creation` スキルに従う
- PR に言及する時はクリッカブルな URL を示す。複数PRが並行する時は各PRの状態(Draft / Open / CI / レビュー)を能動的に確認して伝える

## 破壊的・外部影響操作の確認

- git/gh の破壊的操作(force push / `--no-verify` / ブランチ削除 / PRの close・comment・merge・ready・review / 非Draft PR作成)は hook が確認プロンプトを強制する
- hook が検知できない操作(外部チャネルへの投稿・状態変更、既存ファイルの大量削除・一括リネーム)も「文面提示 → ユーザー承認 → 実行」を経る
- 「ロールバックの急ぎ」「CIを通すための一時回避」を理由に独断実行しない。回避策が必要ならまずユーザーに確認する

## レビューゲート

- 対話セッションで設計/プランファイルを生成・更新したら、実装前に crit レビューを通す(順序: 設計の crit → 実装 → `superpowers:requesting-code-review` → Draft PR)。headless 自走時は Draft PR を人間レビューのゲートとする
- crit は `~/.claude/crit-review.sh <file>`(Bash timeout 600000ms)で起動し、完了後に `crit status <file>` のコメントを反映。スクリプトが tmux / herdr を自動判別して隣ペインに TUI を開く。どちらでもない環境ではスクリプトが失敗するので、ブロッキング起動せず手動実行をユーザーに依頼する。TUI 表示中は対象ファイルを編集しない
- 可読性を下げる指摘(否定形への反転、抽象化の追加、将来仮定への対応)はそのまま適用せず、該当箇所・指摘要約・代替案(適用しない案を含む)を添えてユーザーの判断を仰ぐ
- 「CIが通った」「レビューエージェントが承認した」を設計判断の正当性の根拠としない。指摘への反論コメントの投稿は文面の承認を得てから

## 実装の方針

- 最小スコープで作る(YAGNI)。投機的な汎用化・抽象化はしない。具体的な将来要件が明示された時のみ拡張設計に踏み込む
- 既存実装とリポジトリの慣習を踏襲する。着手前に類似実装を探し、同じパターンで書く
- エラーは抑制(`@ts-ignore`、try-catchの握りつぶし)せず根本原因を修正する。関連が薄く見えるエラーも必ず解決する
- 技術的制約により設計(How)を変更したら、PR作成前に設計ドキュメントを実装に合わせて更新する(What は変えない)。乖離が残る場合はPR説明に理由を記載する

## ドキュメンテーション(How / What / Why / Why not)

- コードには How(コード自身で表現)、テストコードには What(仕様が読めるテスト名)、コミットログには Why、コードコメントには Why not(非自明な制約・前提・落とし穴)を書く
- 処理を逐語的になぞるコメント、型・変数名から自明な説明は書かない。PR本文・コミットは diff で分かる「何を」を最小化し「なぜ」に絞る。1項目は1短文で言い切る

## セキュリティ

- APIキー、パスワード等は環境変数で管理(ハードコード禁止)
- 信頼境界を越える値(ユーザー入力・Cookie・ヘッダー・外部APIレスポンス・Webhook)は検証を通るまで信頼しない。検証・認可・依存先の失敗は拒否側に倒す(fail-close)
- 認可は認証と区別し、主体×リソース×操作×コンテキストで確認する。IDやURLを知っていることを許可の根拠にしない
- 日常の脆弱性検出は `security-guidance` plugin が自動実行する。能動的なチェック(境界レビュー・脅威モデリング・認可レビュー・依存スキャン)は `/security-hardening` スキルを使用

## ユーザーとの対話・タスク進行

- Goal / Constraints / Acceptance criteria が揃っていればそれに従い、揃っていなくても細かい手順確認は最小限にとどめ自律的に進める
- 選択肢は A / B / C 形式で推奨案を添えて提示する。質問は番号付きでまとめて聞き、小出しの質問攻めをしない
- 作業を停止する時は現在の状態を明示し、承認を得たら即着手する。長時間処理は途中経過を逐次共有する

## Subagent の使用指針

- 複数ファイルへの並列処理、互いに独立した複数タスクで積極活用する。それ以外の不要な呼び出しは避ける(過剰な委譲は性能低下のリスク)
- サブタスクの委譲は同期実行で結果を待ってから続行する。「完了通知を待つ」と述べてターンを終えない
