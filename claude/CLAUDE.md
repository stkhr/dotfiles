# AI駆動開発 共通ガイドライン

思考は英語で行い、最終的な出力は必ず日本語で提供してください。

## 最優先原則（事実性・先頭厳守）

- 実行していない作業を完了として報告しない。ファイル変更・commit・push・PR作成・外部API/AWSの結果は、**実行直後に実出力（`git status -sb` / `gh pr view` / コマンドのJSON出力）で確認できたものだけ**「完了/成功」と書く。確認できないものは「未確認」と明示する
- ツールの出力が空・壊れている時に、結果を推測で埋めない。再実行して実出力を得てから報告する
- worktree はリポジトリ内（`.worktrees/<branch>`）に作成する。ファイル生成・依存解決系コマンド（`uv` / `pip` / `npm` / `pytest` / `mkdir` 等）の前に `pwd` で現在地を確認し、worktree 配下であることを保証する

## Git Workflow

- コンベンショナルコミット形式を使用: `feat:`, `fix:`, `docs:`, `test:`, `refactor:`, `chore:`
- コミットメッセージは英語で記述
- PRのタイトル・本文は日本語で記述
- **CRITICAL - MANDATORY RULE**: main/masterブランチへの直接コミットは絶対に禁止。ユーザーが明示的に「mainに直接コミットしてよい」と指示した場合のみ例外とする
- コミット前に必ず `git branch --show-current` で現在のブランチを確認すること
- main/masterにいる場合は、まずフィーチャーブランチを作成してからコミットすること

## PR作成の方針

GitHubリポジトリでの作業依頼に対しては、以下を既定動作とする:

- **コミットを伴う依頼は常にPRまで作成する**（typoや1行修正、ドキュメント・設定ファイルの変更であっても、フィーチャーブランチを切ってPR化する）。ユーザーが明示的に「直接コミットしてよい」「PR不要」と指示した場合のみ例外
- **リポジトリ側でPRフォーマットが指定されている場合は、それに従うことを最優先する**
  - 確認対象（上から順に優先）:
    1. `.github/PULL_REQUEST_TEMPLATE.md` / `.github/pull_request_template.md` / `PULL_REQUEST_TEMPLATE.md` / `docs/PULL_REQUEST_TEMPLATE.md`
    2. `.github/PULL_REQUEST_TEMPLATE/` ディレクトリ配下の複数テンプレート（変更種別に合うものを選択）
    3. `CONTRIBUTING.md` 内のPR作成ガイドライン（フォーマット指定や記入ルールがあればそれに従う）
  - 存在する場合は項目を勝手に省略・追加せず、フォーマット構造を保ったまま記入する
- 上記いずれも無い場合は、以下のシンプル構成を既定とする。Test plan等の追加セクションは作らない:

  ```
  ## Summary
  - <変更点1>
  - <変更点2>
  ```

  - 箇条書きは2〜4個程度、各行は1文で簡潔に
  - 「なぜ」を中心に書き、「何を」はdiffで明確な部分は最小限に
- PR本文末尾に `🤖 Generated with [Claude Code]` 等のClaude Code署名行は**付けない**
- **委譲を伴う実装タスクの既定の終点は Draft PR とする**。次の順で自走し、PR作成前のユーザー承認は挟まない:
  1. `superpowers:requesting-code-review` を実行し、Critical な指摘は修正する
  2. `gh pr create --draft` で Draft PR を作成する
  3. 作成後に PR URL・本文要旨・残レビュー所見をユーザーに共有する
- Draft PR の本文は上記の `## Summary` に加えて `## レビュー所見` セクションを設け、Critical（修正済み）/ Important・Minor（未対応・要判断）を列挙する。これにより PR description とコードの指摘が PR 上で1か所にまとまる
- Draft → Ready 変換、非Draft（Ready）PR の作成、PR のマージ・クローズ・コメント投稿は従来どおり「破壊的・外部影響操作の確認ハードゲート」の対象（文面提示 → 承認 → 実行）

## 破壊的・外部影響操作の確認ハードゲート

以下の操作は、実行前に必ず「文面提示 → ユーザー承認 → 実行」の3段階を経ること。ユーザーが事前に明示的に許可した場合のみ例外とする。委譲型タスク（後述）であっても、本セクションの操作は例外的に確認を挟むこと。

- PR の **クローズ / コメント投稿**（Copilotレビューへの反論コメントを含む）/ **非Draft（Ready）PR の作成** / **Draft → Ready 変換**
  - 例外: **Draft PR の作成（`gh pr create --draft`）は本ゲートの対象外**。委譲タスクの終点として事前承認なしで作成してよい（作成後に PR URL・本文要旨を共有する）。詳細は「PR作成の方針」を参照
- ブランチ削除 / force push
- `--no-verify` による pre-commit/pre-push hook のスキップ
- 外部チャネル（Slack / JIRA / Confluence 等）への投稿・チケット状態変更
- 既存ファイルの大量削除 / 一括リネーム

「ロールバックの急ぎ」「CIを通すための一時回避」を理由に独断実行しない。回避策が必要ならまずユーザーに確認する。

## GitHub CLI (`gh`)

GitHub関連の操作はすべて `gh` コマンドを使用してください（MCPサーバーは使用しない）。

## CI/CD

- PRを作成する前に、必ずローカルでテスト・ビルド・lint/format を実行
- **PR作成前に必ず `superpowers:requesting-code-review` スキルを実行し、独立したコードレビューエージェントの承認を得ること（自己評価での代替不可）**
  - Critical な指摘がある場合は修正してからレビューを再実行する
  - Draft PR フローでは Important・Minor な指摘は Draft PR 本文の「レビュー所見」に記載し、人間レビューでの判断に委ねる。非Draft（Ready）PR を作成する場合は Important な指摘も修正してから作成する
- PR作成依頼を受けたら、まずリポジトリのCI/CD設定（GitHub Actions、CircleCI、その他）の有無を確認する
- CI/CDが設定されている場合、PR作成後は `gh pr checks` でステータスを確認するまでが作業完了
- CI失敗時は即座に修正（放置・無視は禁止）

## crit によるローカルレビュー（設計・コード）

`crit` は AI 生成の設計/プラン・コード差分を tmux 上の TUI でインラインレビューし、コメントを JSON で受け取って反映するツール。Draft PR の手前に、手元の人間レビューを差し込むために使う。必須の `superpowers:requesting-code-review`（AI レビュー）を置き換えない（順序: 設計の crit レビュー → 実装 → requesting-code-review → Draft PR）。

- **契機**: 対話セッションで設計/プランファイル（plan/design 等の Markdown）を生成・更新したら、実装や Draft PR 化へ進む前に crit レビューを通す
- **tmux 内**: `crit review <file> --detach --wait`（人間のレビュー完了までブロック。Bash timeout は 600000ms を指定）→ `crit status <file>` の JSON コメントを Edit で反映 → 必要なら再レビュー
- **tmux 外 / headless**: ブロッキング起動は使わず、`crit review <file>` の手動実行をユーザーに依頼し、完了確認後に `crit status` を反映
- **コード差分**（任意）: `crit review --code --detach --wait` → `crit status --code`
- TUI を開いている間は対象ファイルを編集しない。コメントは `.crit/`（global gitignore 済み）に保存される

## コードレビュー指摘の受け止め方

レビュー（人間 / Copilot / 自動レビューエージェント問わず）の指摘を取り込む際は、以下を守ること:

- **可読性を下げる方向の指摘**（特に条件式の否定形への反転、抽象化の追加、将来仮定への対応）はそのまま適用しない。該当箇所・指摘要約・代替案（適用しない案を含む）を提示してユーザーの判断を仰ぐ
- **「CIが通った」「レビューエージェントが承認した」を、設計判断の正当性の根拠としない**。これらは弱い根拠であり、人間の判断を上書きしてはいけない
- 指摘への反論コメントをPR/Issueに投稿する場合は、文面をユーザーに提示して承認を得てから投稿する

## 設計と実装の一貫性

- 実装中に技術的制約により設計（How）を変更した場合、PR作成前に設計ドキュメントを実装に合わせて更新する
- 変更してよいのはHow（実現方法）のみ。What（実現したいこと）は変えない
- 設計ドキュメントと実装に乖離がある場合はPRの説明に変更理由を記載する

## エラーハンドリング

- エラーの抑制（`@ts-ignore`、try-catchで握りつぶす等）ではなく、根本原因を修正
- 関連が薄く見えるエラーでも必ず解決する

## 外部API・AWSサービス実装の事前確認

AWS / DataDog / Auth0 / Slack 等の外部APIや、AWSサービス設定を扱うコードを書く前に、必ず一次情報（公式ドキュメント or 実機のCLI/APIレスポンス）で以下2軸を確認すること:

- **識別子と命名規則**: IAM action prefix・サービス識別子・suffix・正規表現制約（例: `aidevops:*` を `devops-agent:*` と推測しない / Service Connect `dnsName` の `.internal` 有無は `describe-services` で確認）
- **スキーマと制約値**: リクエスト/レスポンスJSON構造・必須フィールド・AWS推奨設定との整合（例: ALB `idle_timeout` ≥ KeepAliveTimeout）

「実装 → 実行時エラー → 段階的修正」は短期的に速く見えても、ユーザーのレビュー負荷を確実に増やす。実装着手前の確認を優先する。

### 仮説の検証ルール

- 同じ問題で仮説が3回外れたら、レイヤを1つ上げて疑う（ミドルウェア → アプリ → DB / ネットワーク → OS）
- 推測のまま設定値・コード値を確定させない。実機検証（CLIワンライナーをユーザーに提供して結果を貰う等）を挟む

## コメント・記述の簡潔さ（デフォルト）

簡潔さを既定とし、説明を増やすのは明示的に求められた時だけ。冗長な説明・自明な記述はデフォルトで省く。

- **コードコメント**: コードを読めば分かることは書かない。コメントするのは「なぜそうするか」「非自明な制約・前提・落とし穴」のみ。処理を逐語的になぞるコメント、型・変数名から自明な説明は書かない
- **PR本文・コミットメッセージ**: diffで分かる「何を」は最小限にし「なぜ」に絞る（「PR作成の方針」参照）。前置き・定型句の繰り返しを避ける

## セキュリティ

- APIキー、パスワード等は環境変数で管理（ハードコード禁止）
- 日常の脆弱性検出は `security-guidance` plugin が自動実行する（per-edit / end-of-turn / commit の3層レビュー）
- 能動的なセキュリティ強化（設計レビュー、脅威モデリング、依存関係スキャン、総点検レポート）は `/security-hardening` スキルを使用

## タスク指示のフォーマット（委譲型）

ユーザーからタスクを受けた際、以下の要素が揃っている場合はそれに従って動作すること。揃っていない場合でも、細かい手順確認は最小限にとどめ自律的に判断して進めること:

- **Goal**: 達成すること
- **Constraints**: 守るべき制約
- **Acceptance criteria**: 完了の判定基準

途中介入を求めず、委譲されたものとして自律的に進めること。コミットを伴う実装タスクの終点は原則 Draft PR の作成とリンク共有とする（「PR作成の方針」参照）。
- 例外: 人間が在席する**対話セッション**で設計/プランファイルを生成した場合のみ、Draft PR 化の前に crit によるローカルレビューを差し込んでよい（「crit によるローカルレビュー（設計・コード）」参照）。headless で自走する場合は従来どおり Draft PR を人間レビューのゲートとし、crit のブロッキング起動は行わない

## Subagent の使用指針

以下の場合に Subagent を積極的に活用すること:
- 複数ファイルへの並列処理（fanning out across files）
- 互いに独立した複数タスク（independent items）

上記以外では不要な Subagent の呼び出しを避けること。過剰な委譲指示は性能低下のリスクがある。

## Terraform スタイル規範

Terraform コードを書く際は、以下のスタイルに従うこと:

- `count` / `for_each` の三項演算子は**肯定形**を使う。否定形は読み手の脳の負担を増やすため避ける
  - OK: `count = var.env_name == "stg" ? 1 : 0`
  - NG: `count = var.env_name != "prd" ? 1 : 0`
- `terraform fmt` は変更ファイル単位で実行する（`-recursive` でスコープ外まで整形しない）

## Compaction preferences

コンテキスト圧縮時は、以下を必ず保持:
- 変更したファイルの一覧
- 実行したテストコマンドとその結果
- 未完了のタスク
