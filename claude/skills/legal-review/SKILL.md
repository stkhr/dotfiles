# Legal Review Skill

## Purpose

プロダクトの機能を法令のトリガー条件と突合し、**法務案件の可能性がある箇所を検知して
アラートを出す**ためのスキル。目的はアラート検知とエスカレーション判断の支援であり、
法的判断の代替ではない。

前提となる考え方:

- 法的リスクは「条文の解釈」より先に「機能と法令名の紐づけ」で見えるようになる。
  仕様に特定のキーワード(ポイント・DM・解約・スクレイピング等)が現れた時点で
  疑うべき法令を挙げられることがゴール
- 実装後に発覚した法務案件は機能の作り直しになる。レビューは仕様・設計段階が最も効く
- 「該当なし」の判断も根拠つきで記録する。後で仕様が変わった時に再判断の起点になる

## When to Use

- 「法務レビューして」「法務チェック」「リーガルチェック」という明示的な指示
- 新機能の仕様・設計ドキュメントに対する法務観点の確認依頼
- リリース前の法務セルフチェック
- 「この機能って法律的に大丈夫?」という個別の相談

## When NOT to Use

- PR ごとの自動レビュー(本スキルはユーザーの任意タイミングでの明示起動のみ)
- SES・外注・偽装請負・下請法・フリーランス保護新法の確認(取引/組織の法務は対象外)
- 経理・会計処理と DB 設計の整合性の確認(法務ではなく設計指針の問題)
- 法的判断・条文解釈の最終確定(必ず弁護士・法務担当者へ)

## Workflow

1. **対象の把握** — 渡された仕様・設計ドキュメントを読む。対象の指定がなければ
   コードベースから機能を抽出する(ルーティング・画面・決済処理・外部連携・
   バッチ処理を手がかりに)
2. **機能タイプの抽出** — 対象を次の 8 タイプに分類する。1 機能が複数タイプに
   該当することもある
   - 決済・ポイント(付与/チャージ/送金/換金)
   - メッセージング(DM/チャット/通報・監視)
   - 退会・データ削除
   - 外部タグ・SDK・API 連携(データの外部送信)
   - 素材・ライセンス(画像/フォント/OSS/UGC/AI 生成物)
   - 購入・解約フロー(EC/サブスク)
   - スクレイピング・データ収集
   - 販売商材(何を売るか)
3. **逆引き突合** — [reference.md](reference.md) のトリガー表と突合し、
   該当するアラートを列挙する
4. **鮮度確認** — 検知した法令について WebSearch で省庁等の一次情報を確認する。
   reference.md は執筆時点のスナップショットであり、法令は改正される。
   数値基準(供託基準額・有効期限・支払期日等)と施行状況は必ず現行情報で裏取りする
5. **レポート出力** — 下記フォーマットで出力する
6. **免責の明記** — レポート末尾に免責と「専門家相談の準備情報」を必ず添える

## Report Format

```markdown
# 法務レビュー結果: <対象>

## アラート一覧

| 機能 | 疑うべき法令 | 確認事項 | 推奨アクション |
|------|------------|---------|--------------|
| ... | ... | ... | 仕様変更の選択肢 / 専門家相談の要否 |

## 検討したが該当なしと判断した項目

| 機能タイプ | 判断根拠 |
|-----------|---------|
| ... | ... |

## 専門家に相談する場合の準備情報

- 機能の概要(何を実装しようとしているか)
- お金の動き(金額・流れ・換金性の有無)
- データの送信先(外部サービス連携の有無)
- 実装の進捗とリリース予定日

## 免責

本レビューは法務案件の可能性を検知するためのものであり、法的アドバイスではありません。
アラートの最終判断は必ず弁護士・法務担当者に相談してください。
法令は改正されるため、数値基準・施行状況は各省庁の一次情報を確認してください。
```

推奨アクションは具体的に書く。「法務に相談」だけでなく、リスクを下げる設計上の
選択肢(例: 有効期限の設定・対価スキームへの組み替え・閲覧権限の制限)があれば併記し、
その選択がビジネス判断を含む場合はその旨を明示する。
