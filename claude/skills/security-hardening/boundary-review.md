# 境界レビュー チェックリスト

セキュリティチェックを実行するための質問集。設計レビュー・コードレビュー・
AIエージェントへの指示にそのまま使える粒度で書いてある。

## 保護対象の整理

チェックの前に、何を守るかを特定する:

| 資産 | 例 | 損なわれると |
| --- | --- | --- |
| 機密データ | 個人情報、決済情報、認証トークン、社内情報 | 漏洩(Confidentiality) |
| 状態を変える操作 | 決済、権限付与、データ削除、設定の書き換え | 不正実行(Integrity) |
| 信頼関係 | 依存ライブラリ、外部API、CI/CD、連携サービス | 侵害の波及・改ざん |

## 基本の3観点

信頼境界を越えるデータフロー・処理ごとに確認する:

1. **出所(provenance)** — この値は信頼境界の外から来ていないか。
   ユーザー入力、Cookie・ヘッダー、外部APIレスポンス、Webhook、キューのメッセージは
   検証を通るまで信頼しない
2. **実行権限(authority)** — この処理はどの主体の権限で動くか。その主体は
   対象リソースへのこの操作を許可されているか。認証(本人確認)と認可(権限確認)を
   区別して確認する
3. **失敗時の挙動(failure mode)** — 検証・認可・依存先が失敗したとき、
   拒否側に倒れるか(fail-close / deny by default)。エラーで内部情報を漏らさないか

## リクエストライフサイクルに沿った確認順

チェック漏れを防ぐため、リクエストの流れに沿って順に確認する:

1. **エンドポイント** — このエンドポイントは棚卸しされているか。想定外の経路(検証用、旧版、Webhook)から到達できないか
2. **認証** — 呼び出し元は誰か。トークン・セッションの検証、期限、失効
3. **認可** — その主体はこのリソースへのこの操作を許可されているか
4. **入力検証** — ペイロードはスキーマに適合するか。想定外のフィールドを拒否するか
5. **処理・出力** — 出力先の文脈に応じたエンコード。返却フィールドは必要最小限か
6. **記録** — 誰が・いつ・何をしたかを監査ログに残すか

## 脆弱性別チェック観点

| 脆弱性 | 確認すること |
| --- | --- |
| XSS | 出力先の文脈(HTML本文・属性・URL・JS)ごとのエンコードがあるか。CSPで実行範囲を制限しているか |
| Injection | クエリ・コマンドがパラメータ化されているか。ユーザー由来の値を文字列連結で命令に埋め込んでいないか |
| CSRF | Cookie認証の状態変更エンドポイントに CSRFトークン / SameSite / Origin検証があるか |
| IDOR / BOLA | リソースIDを別ユーザーのものに差し替えたリクエストが所有者・テナント検証で拒否されるか。推測困難なID(UUID)を認可チェックの代替にしていないか |
| BFLA | 管理者用・内部用エンドポイントにサーバー側のロール検証があるか(UIに出していないことは制御ではない) |
| マスアサインメント | リクエストボディをそのまま更新処理へ渡していないか。許可フィールドの明示(allowlist)があるか |
| SSRF | ユーザー指定URLのスキーム・ホストを検証しているか。内部ネットワーク・クラウドメタデータAPIへの到達を遮断しているか |
| 設定ミス | CORS、Cookie属性、CSP、エラー出力、デバッグ設定が全環境で意図通りか |
| サプライチェーン | lockfileによる固定、依存の由来確認、CI/CD権限の最小化、SBOMがあるか |
| 監査不備 | 認証失敗・認可失敗・状態変更が記録され、リクエスト・ユーザー・テナントの識別子で追跡できるか |

## 前提にする攻撃者モデル

- **認証済みユーザーを必ず含める**。アクセス制御の欠陥の多くは正規アカウントからの
  権限昇格で悪用される
  - 水平方向の権限昇格: 同ロールの別ユーザー・別テナントのリソースへアクセスする
  - 垂直方向の権限昇格: 一般ユーザーが管理者向け機能を実行する
- **クライアント側の制御はバイパスされる前提**で見る。攻撃者はUIを経由せず
  APIを直接呼ぶ(開発者ツール・API仕様・パス推測)。フロントエンドの表示制御・
  バリデーションはUXであって防御ではない
- **個々には正当なリクエストの悪用**を考える。不正な文字列だけでなく、正当な形式の
  リクエストを想定外の順序・回数・組み合わせで送る(ビジネスロジックの悪用)

## クライアント実行環境の確認

ブラウザの挙動はHTTP・DOM・Cookie・オリジンの各仕様で決まる。実装側の想定と、
仕様上クライアントに可能な操作とのずれを確認する:

| 仕様上の挙動 | 悪用パターン | 確認する制御 |
| --- | --- | --- |
| 条件を満たすCookieは自動で送られる | CSRF | SameSite属性、CSRFトークン、Origin検証 |
| 文字列がHTML/JSとして解釈され得る | XSS | 文脈別エンコード、CSP、危険API(`innerHTML`等)の回避 |
| クロスオリジン制御はヘッダーで緩められる | CORS誤設定による読み取り | 許可オリジンの限定、credentialsの扱い |
| トークン・状態がクライアントに残る | トークン窃取、失効漏れ | 保存場所、有効期限、ログアウト・権限変更時の失効 |

## 認可モデルの確認

「認可チェックがある」だけでは不十分。ABACの要素に分解して確認する:

| 要素 | 確認すること |
| --- | --- |
| サブジェクト | リクエストの主体が正しく識別・検証されているか |
| リソース | 対象リソースの所有者・所属テナントを検証しているか |
| アクション | 読み取り・作成・更新・削除それぞれに同じ強度の検証があるか |
| コンテキスト | テナント・組織・リソースの状態(例: 確定済みデータの変更禁止)を考慮しているか |

- 認可ロジックはハンドラごとに書かず一元化する(middleware / policy 層、deny by default)
- IDを知っていること・URLに到達できることを許可の根拠にしない

## フェーズ別チェック

### 設計

| 確認事項 | 成果物 |
| --- | --- |
| 信頼境界はどこか。境界の外から来るデータは何か | データフロー図、脅威モデル |
| 認可モデル(ロール・所有者・テナント × 操作)が定義されているか | 認可マトリクス |
| 障害・例外時のデフォルト動作が拒否側に定義されているか | 設計ドキュメント、ADR |
| CIで機械的に検証する条件は何か | スキーマ、lint・scanルール |

### 実装

| 確認事項 | 見る場所 |
| --- | --- |
| 入力がスキーマで検証され、未定義フィールドが拒否されるか | DTO、OpenAPI、バリデーション層 |
| レスポンスが必要最小限のフィールドに絞られているか | レスポンスモデル、シリアライザ |
| 認可チェックが一元化された層を通っているか | middleware、policy、decorator |
| 生成コードの差分に検証漏れ・依存追加・エラー握りつぶしがないか | diff全体 |

### テスト

| 確認事項 | テストの書き方 |
| --- | --- |
| 水平方向の権限昇格が拒否されるか | 別ユーザーの認証情報で他者のリソースIDを指定する |
| 垂直方向の権限昇格が拒否されるか | 一般ロールで管理者用エンドポイントを呼ぶ |
| 未定義フィールドが無視・拒否されるか | 権限・価格・所属などの内部フィールドをペイロードに混入させる |
| 異常系・境界値・並行実行がカバーされているか | 正常系と同じ操作の「拒否されるべきケース」を対で書く |

### 運用

| 確認事項 | 手段 |
| --- | --- |
| 公開エンドポイントの棚卸しができているか | API仕様、ゲートウェイ・アクセスログ、廃止管理 |
| セキュリティイベントを検知できるか | 認証・認可失敗、レート制限到達、異常なエラー率の監視 |
| セキュリティルール(CI設定・エージェント向け指示)が現状と一致しているか | 設計変更・インシデント後の見直し |
| インシデント時に行為を追跡できるか | リクエストID・ユーザーID・テナントIDを含む監査ログ |

## 不変条件として残すもの

チェック結果は使い捨てにせず、レビューで読めて・CIで検証できて・例外を追跡できる形で残す:

| 成果物 | 内容 |
| --- | --- |
| **認可マトリクス** | ロール・所有者・テナント × 操作の許可一覧 |
| **入力契約** | スキーマ、型・範囲、未定義フィールドの扱い |
| **出力契約** | 返却してよいフィールド、PIIの扱い、エラー応答の形式 |
| **監査・リリース条件** | 記録するイベント、スキャンの通過条件、例外承認の記録 |

不変条件の書き方の例:
- テナントIDが一致しないリソースへのアクセスは全操作で拒否する
- 状態を変更する操作は必ず監査イベントを発行する
- スキーマに定義のないフィールドを含むリクエストは受理しない

## 多層防御の配置

どの層で何を判断するかを設計する。層ごとに得意な判断が違う:

- **クライアント**: UX目的の入力補助。防御としては数えない
- **エッジ(WAF・ゲートウェイ・CDN)**: 認証前の遮断、レート制限、既知パターンの粗いフィルタ
- **アプリケーション**: 認可(所有者・ロール・テナント・状態)、入出力契約 — 業務的な意味の判断はここでしかできない
- **データ層**: 最小権限の接続、行レベルの制約
- **運用**: 監視・検知・棚卸し・依存更新

## 自動化と手動レビューの分担

- **ツール(SAST / DAST / dependency scan / secret scan)に任せる**: 危険な関数、
  既知CVE、ヘッダー不足、秘密情報の混入などパターン化できる検出
- **このチェックリストで見る**: 所有権・業務フロー・状態に応じた許可など、ドメイン固有の
  ルールに依存する判断。正規の認証済みリクエストで悪用できる欠陥はツールでは検出しづらい

## 導入の優先順位

一度に全部やらない。悪用時の影響が大きく、かつ自動検出しづらいものから:

1. **認可マトリクスを書く** — 許可の全体像をレビュー可能にする
2. **入力スキーマをCIで強制する** — 未定義フィールド拒否・上限値を機械検証に落とす
3. **権限昇格のテストを追加する** — 正常系と対になる「拒否されるべきケース」を常設する
