# PDM Assist Reference — プロダクトマネジメントのフレームワーク集

SKILL.mdのワークフローを補完する詳細リファレンス。必要な箇所だけ参照する。
定番フレームワーク（Torres の継続的発見、Savoia のプレトタイピング、Maurya の
Running Lean、Dunford のポジショニング、Fitzpatrick の The Mom Test）を
個人事業主スケールに要約したもの。

## 前提（アサンプション）マッピング

アイデアは前提の束。3分類で洗い出す:

| 分類 | 問い |
|------|------|
| Desirability（欲しいか） | その課題は実在するか。顧客は今どう解決しているか。乗り換えるほど困っているか |
| Viability（商売になるか） | 払う人がいるか。いくらなら払うか。続けて払うか。届けるコストを下回らないか |
| Feasibility（作れるか） | 自分のスキル・時間・資金で作り切れるか。作った後も運用し続けられるか |

洗い出した前提を **致命度（崩れたら終わりか）× 確証のなさ（証拠がないか）** で
評価し、「致命度が高く確証がない」ものから検証する。

## 安い実験カタログ（プレトタイピング）

作り込む前に使う。上ほど安い。

| 実験 | やり方 | 測るもの |
|------|--------|---------|
| 顧客インタビュー | 想定顧客5人に過去の行動を聞く | 課題の実在、現在の代替手段 |
| ランディングページ | 価値提案＋登録フォームだけ作って流す | 登録率 |
| フェイクドア | 「近日公開」ボタンへの反応を測る | クリック率 |
| コンシェルジュ | プロダクトの代わりに手作業でサービス提供 | 対価を払うか、繰り返し使うか |
| オズの魔法使い | 表側だけ作り、裏は手動で回す | 実際の利用行動 |
| 先行販売 | 作る前に売る（予約・前払い） | 実際のお金（最強の証拠） |

証拠の強さ: **意見 < 連絡先の提供 < 時間を使う < お金を払う**。
顧客が何かを差し出したものだけを証拠と数える。

## 顧客インタビューの要点（The Mom Test）

- 未来の意向（「使いますか?」）ではなく **過去の行動**
  （「最後にその問題が起きた時どうしましたか?」）を聞く
- 自分のアイデアを説明しない・売り込まない。相手の仕事や生活を聞く
- 褒め言葉・一般論・仮定の話は証拠にならない。具体的な過去とお金の話だけメモする
- 5人程度で傾向は見える。共通の課題が出ないなら、セグメントか課題設定を見直す

## リーンキャンバス

1枚でビジネスモデルを整理する。埋める順番:

1. **課題**（上位3つ）と既存の代替手段
2. **顧客セグメント**（ビーチヘッドを明記）
3. **独自の価値提案**（1文）
4. **ソリューション**（課題に対応する最小限）
5. **チャネル**（顧客への到達経路）
6. **収益の流れ** / 7. **コスト構造**
8. **主要指標**（今のステージで見る数字1〜2個）
9. **圧倒的な優位性**（真似されにくいもの。なければ「なし」と正直に書く）

課題と顧客が先、ソリューションは後。自信を持って埋められない箱は前提として
検証リストに送る。

## ビーチヘッド顧客の絞り込み

最初に狙うセグメントを1つだけ選ぶ。基準:

- **切実さ**: 課題が痛い（「あれば便利」ではない）
- **到達しやすさ**: 自分がリーチできる（コミュニティ・人脈・チャネルがある）
- **支払い能力と意思**
- **口コミの繋がり**: セグメント内で互いに参照し合う

全部を満たす候補がなければ「切実さ × 到達しやすさ」を優先する。

## 値付け

### 3類型

| 方式 | 決め方 | 使いどころ |
|------|--------|-----------|
| コストベース | 原価＋利益 | 最低ラインの確認のみ。これで決めない |
| 競合ベース | 代替手段の価格を参照 | 相場の把握。価格競争に引きずられない |
| 価値ベース | 顧客が得る価値から逆算 | 原則これ。顧客の節約額・売上増を根拠にする |

### 個人事業主が陥りがちな安売りパターン

- 時給換算（価値ではなく作業時間で決める）
- 遠慮値引き（「高い」と言われる前に自分から下げる）
- 無料の既定化（タダに慣れた顧客は有料に転換しない）

対策: 松竹梅の3段階提示 / 想定より1段高く出して反応を見る /
値上げは既存顧客より新規顧客で試す。

## ポジショニング（5要素）

順番に埋める:

1. **競合代替**: 顧客はこれがないと何を使うか（「Excelでやる」「何もしない」も競合）
2. **独自の能力**: 代替にない自分の強み
3. **価値**: その能力が顧客に生む具体的な価値
4. **最も気にする顧客**: その価値が一番刺さる人（ビーチヘッドと一致するか確認）
5. **市場カテゴリ**: 顧客の頭の中のどの棚に置いてもらうか

価値提案文のフォーマット:
**「{顧客}が{課題／ジョブ}する時、{プロダクト}は{競合代替}と違って{独自の価値}を提供する」**

## 運用期: フィードバックの捌き方

- 要望はそのまま実装しない。「なぜそれが欲しいのか」を1回掘って課題に翻訳する
- 声の大きい1人と静かな多数を区別する（同じ課題が複数の出所から来ているか）
- 改善候補の優先順位付けは pm-assist の reference.md
  （Impact × Effort / RICE）を流用する

## アンチパターン（この Skill が避けるべき振る舞い）

- 検証を飛ばして作り込みの計画を立てる（最も高くつく失敗を補佐しない）
- 「使いたい」の声を証拠として扱う（お金・時間・連絡先だけが証拠）
- 全員向けの訴求を作る（ビーチヘッドの放棄）
- 大企業向けフレームワークの持ち込み（五力分析・PESTLE・詳細ペルソナは過剰）
- 競合調査のしすぎ（競合代替はポジショニングの入力に使うだけでよい）
- 安売りの追認（時給換算・遠慮値引きに根拠を求める）
