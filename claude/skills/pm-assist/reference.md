# PM Assist Reference — PMBOKベースの知識体系

SKILL.mdのワークフローを補完する詳細リファレンス。必要な箇所だけ参照する。

## PMBOK第7版 12の原則（行動指針として使う）

1. **スチュワードシップ**: 誠実に、預かったリソースに責任を持つ
2. **チーム**: 協働的なチーム環境を作る
3. **ステークホルダー**: 効果的に関与させる
4. **価値**: 成果物ではなく価値にフォーカスする
5. **システム思考**: プロジェクトを取り巻く相互作用を認識する
6. **リーダーシップ**: 状況に応じたリーダーシップを発揮する
7. **テーラリング**: 状況に合わせてアプローチを調整する
8. **品質**: プロセスと成果物に品質を組み込む
9. **複雑性**: 複雑さに気づき、対処する
10. **リスク**: 脅威と好機の両方に対応する
11. **適応力と回復力**: 変化に適応する
12. **チェンジマネジメント**: 望ましい将来の状態への移行を可能にする

この Skill で特に重要なのは **4（価値）・7（テーラリング）・10（リスク）**。

## 知識エリア別チェックリスト（PMBOK第6版の10知識エリア）

タスク洗い出し時の網羅性チェックに使う。全部やるのではなく、抜けを見つける道具。

| 知識エリア | 洗い出し時の問い |
|-----------|----------------|
| 統合 | 変更が起きた時、誰がどう判断するか決まっているか |
| スコープ | やらないこと（アウトオブスコープ）が明文化されているか |
| スケジュール | 依存関係とクリティカルパスが見えているか。バッファはあるか |
| コスト | 予算の上限と、超えそうな時のエスカレーション先は明確か |
| 品質 | 完了の定義・受け入れ基準はあるか。誰が検収するか |
| 資源 | 必要なスキル・人・環境・ライセンスは確保できているか |
| コミュニケーション | 誰に・何を・どの頻度で報告するか決まっているか |
| リスク | リスクの棚卸しと見直しのサイクルはあるか |
| 調達 | 外部契約・発注のリードタイムはスケジュールに入っているか |
| ステークホルダー | 反対しうる人・巻き込み忘れている人はいないか |

軽量型プロジェクトでは スコープ / スケジュール / リスク / ステークホルダー の4つに絞ってよい。

## リスクカテゴリチェックリスト

Step 5 の洗い出しで使う。各カテゴリで「このプロジェクトでは？」と自問する。

### 技術リスク
- 未検証の技術・初めて使うツールへの依存
- 性能・スケーラビリティが本番負荷で未確認
- レガシーシステム・外部システムとの統合の複雑さ
- 技術的負債による想定外の工数増
- データ移行の複雑さ・データ品質の問題

### 外部リスク
- ベンダー・委託先の遅延、品質不良
- 他チーム・他部門の成果物待ち
- 法規制・コンプライアンス要件の変更
- 外部APIやプラットフォームの仕様変更・障害

### 組織リスク
- キーパーソンの離脱・異動・稼働の取り合い
- 優先度変更によるリソースの引き剥がし
- 予算の凍結・削減
- 承認プロセスの長期化（特にエンプラ型）
- 組織変更・体制変更

### プロジェクトマネジメントリスク
- 見積りの精度不足（特に初期見積りへの過度なコミット）
- スコープクリープ（要求の漸増）
- 曖昧な完了基準による手戻り
- コミュニケーション不足による認識ずれ
- 進捗の見える化不足（気づいた時には手遅れ）

### ステークホルダーリスク
- 意思決定者の不在・決定の遅延
- ステークホルダー間の利害対立
- エンドユーザー・現場の巻き込み不足
- 期待値の不一致（できあがってから「思っていたのと違う」）

### 好機（ポジティブリスク）も忘れずに
- 早期リリースで得られるフィードバック
- 再利用できる既存資産の発見
- スコープ削減の交渉余地

## リスク評価と対応方針

### 評価: 発生確率 × 影響度

3段階（高・中・低）で十分。定量化に時間をかけない。

| | 影響:低 | 影響:中 | 影響:高 |
|---|---|---|---|
| **確率:高** | 中 | 高 | **最優先** |
| **確率:中** | 低 | 中 | 高 |
| **確率:低** | 低 | 低 | 中 |

### 対応方針（脅威に対する4類型）

| 方針 | 内容 | 例 |
|------|------|-----|
| 回避 | リスクの原因自体を除去する | 未検証技術をやめて実績ある技術にする |
| 軽減 | 確率か影響を下げる | 早期にスパイク（技術検証）を行う |
| 転嫁 | 第三者に移す | 保守契約・保険・固定価格契約 |
| 受容 | 受け入れて備える | コンティンジェンシー（予備費・予備日）を積む |

「高」以上のリスクには必ず対応タスクを紐付け、タスクリストに反映する。

## ステークホルダー分析（権力 × 関心グリッド）

エンプラ型、またはステークホルダーが5人以上いる場合に使う。

| | 関心:低 | 関心:高 |
|---|---|---|
| **権力:高** | 満足させ続ける（サプライズを起こさない） | 綿密に管理する（密に巻き込む） |
| **権力:低** | 監視する（最小限の労力） | 情報を提供し続ける |

各ステークホルダーについて: 現在の態度（支持 / 中立 / 抵抗）と、望ましい態度、
そのギャップを埋めるアクションを整理する。

## 見積りの技法

- **三点見積り**: (楽観 + 4×最頻 + 悲観) ÷ 6。不確実性が高いタスクに使う
- **類推見積り**: 過去の類似タスクから。実績データがあれば最速
- **相対見積り**: タスク間の相対サイズで並べる。軽量型のチームで有効
- 原則: **不確実性が高い段階の見積りは幅で示す**（例: 2〜4週間）。
  一点見積りをコミットメントとして扱わせない

## フェーズ別の重点成果物

| フェーズ | 軽量型 | エンプラ型 |
|---------|--------|-----------|
| 立ち上げ | ゴール1枚メモ（目的・完了定義・体制・期限） | プロジェクト憲章、ステークホルダー登録簿 |
| 計画 | タスクリスト＋マイルストーン＋リスクトップ5 | WBS、スケジュール、リスク登録簿、コミュニケーション計画 |
| 実行・監視 | 週次の差分確認（完了 / 遅延 / 新リスク） | 進捗レポート、変更管理ログ、リスク登録簿の更新 |
| 終結 | 残作業リスト＋振り返りメモ | 完了報告、教訓（Lessons Learned）、移管ドキュメント |

## アンチパターン（この Skill が避けるべき振る舞い）

- 数人のプロジェクトにWBSやRACIチャートを持ち込む（過剰プロセス）
- リスクを列挙するだけで対応方針とオーナーを付けない（リスク劇場）
- 全タスクを同じ粒度で分解しようとする（遠い将来は粗く、直近は細かく: ローリングウェーブ計画）
- ユーザーへの質問攻めで前に進まない（分かっている範囲で仮組みし、前提として明示する）
- 見積りの一点回答（幅と根拠を示す）
